# DBM Design System ŌĆö Token Spec

Companion to `dbm-tokens/` (the actual generated token files in W3C Design Tokens Community Group format ŌĆö ready to feed into Style Dictionary and, later, the Tokens Studio Figma plugin for design-file sync).

## Architecture: 3 layers

1. **Primitive** (`primitive/*.json`) ŌĆö raw values, no meaning. Color scales, spacing steps, radii, shadows, etc.
2. **Semantic** (`semantic/*.json`) ŌĆö contextual meaning, references primitives via `{color.purple.600}`-style aliases. **One file per theme** (brand ├Ś mode). This is where multi-brand and light/dark actually live ŌĆö the token *names* (`bg.brand`, `text.primary`) stay identical across every theme file; only what they resolve to changes.
3. **Component** (not yet populated) ŌĆö add component-scoped overrides only when a specific component needs to deviate from its semantic default.

Status/neutral tokens (red, amber, green, blue, gray) are **shared across both brand themes** ŌĆö success is always green, danger is always red, regardless of which brand is active. Only the "brand" slots (bg.brand, text.link, border.focus, icon.brand) swap per theme. This is a deliberate consistency principle, not an oversight ŌĆö status meaning shouldn't be reinterpreted by theme.

## What's in each primitive file

| File | Contents |
|---|---|
| `color.json` | 11-step (50ŌĆō950) scales for `purple`, `emerald`, `gray`, `red`, `amber`, `green`, `blue`, plus `neutral.white`/`neutral.black` |
| `typography.json` | Font families (Nunito primary, Lora secondary), weights, fluid font-size scale, line-heights, letter-spacing |
| `spacing.json` | 4px-base numeric scale, `0` through `32` (=128px) |
| `radius.json` | Soft/rounded scale, `none` (0) through `full` (9999px), anchored around 6ŌĆō24px for the "premium SaaS" feel |
| `shadow.json` | Layered elevation shadows, **separate light/dark values** ŌĆö dark-mode shadows are higher-opacity black, not a naive inversion, and should pair with a subtle border on raised surfaces |
| `breakpoint.json` | `sm`(640) ŌåÆ `3xl`(1920, for enterprise dashboard monitors) |
| `motion.json` | Duration scale (120msŌĆō600ms) + 4 easing curves, including an "emphasized" expo-out curve for premium micro-interactions |
| `other.json` | Icon sizes (paired with Phosphor, 12ŌĆō48px), border widths, opacity scale, z-index layering scale |

## Color scale generation

**Re-derived in OKLCH in Phase 4 (2026-07-18)**, replacing the original HSL-based generation. Scales are generated by `packages/tokens/scripts/generate-color-scales.mjs` using `culori` (MIT-licensed, devDependency scoped to `packages/tokens` ŌĆö never shipped to consumers): an 11-step OKLCH lightness curve, shared across all 7 scales, gives perceptually even steps (unlike HSL lightness, which isn't perceptually uniform ŌĆö this is what caused the pre-Phase-4 `text.success` AA failure below). Chroma tapers toward both ends of each scale, gamut-clamped via `culori.clampChroma` so every generated value is a valid, displayable sRGB hex.

- **`purple-600` (`#5548A4`) and `emerald-600` (`#2E8A7D`) stay exact** ŌĆö the two brand-mandated anchors (`01-vision-and-goals.md`'s decisions log). Every other step in those two scales is the shared canonical L curve shifted by a constant offset so it passes exactly through the anchor's own OKLCH `L` at 600, preserving the curve's spacing while still hitting the required color.
- **`gray`, `red`, `amber`, `green`, `blue` regenerate fully** ŌĆö no anchor requirement, so every step (including 600) comes from the canonical curve directly, using each scale's previous 600 value only to pick a matching hue so the scale keeps its color identity (red stays red, etc.).
- Re-run the script whenever a scale needs re-deriving ŌĆö e.g. adding a 3rd brand theme reuses the same anchored-scale logic with a new brand hex.

**Every contrast pairing changed as a result** (75 of 77 primitive values shifted) and was re-verified from scratch ŌĆö see the running log below.

## Fluid typography

Built with the standard "Utopia" clamp() formula: two type scales (ratio 1.2 at 320px viewport, 1.25 at 1440px viewport) interpolated with `clamp(min, preferred, max)`. One deliberate deviation from a textbook fluid scale: **sizes at/below the base step (`xs`, `sm`, `base`) are fixed, not fluid** ŌĆö scaling small UI text down further on large screens adds nothing and risks going under comfortable reading size. Fluid scaling kicks in from `md` upward, where it actually creates a meaningfully improved heading hierarchy across screen sizes.

Note: `font-size-xs` originally computed to ~11px, below the common 12px practical floor for body-adjacent text. **Fixed in Phase 2** ŌĆö floored at `0.75rem` (12px exactly).

## Contrast verification (done, not assumed)

I ran actual WCAG contrast checks on the semantic pairings rather than eyeballing them. Running log, in the order each was found:

**Phase 1 planning:**
- **White text on `emerald-600` (light mode):** 4.16:1 ŌĆö fails normal-text AA (needs 4.5:1). **Fixed** by moving the light-mode `bg.brand` solid-fill token to `emerald-700` (4.55:1, passes).
- **White text on `emerald-500` (dark mode):** 2.39:1 ŌĆö fails badly. **Fixed** by using dark text (`gray-950`) instead of white for `text.on-brand` in the emerald dark theme (6.92:1, passes).

This is why `text.on-brand` is defined per-theme rather than as a single global "white" constant ŌĆö the right foreground color for a solid brand fill depends on how light that specific brand color is, and emerald sits at a very different perceptual lightness than purple even at the same numeric step.

**Phase 3 (building Button/Badge/Avatar surfaced more gaps ŌĆö the token set wasn't as complete as the Phase 1 pass assumed):**
- **No `text.on-danger` existed** (only `text.on-brand`, calibrated for brand colors specifically). Computed directly: white on `red.600` (light themes) = 6.12:1, white on `red.500` (dark themes) = 4.74:1 ŌĆö both pass AA. Added to all 4 themes.
- **No `bg.danger-hover` existed.** Mirroring `bg.brand-hover`'s dark-mode-lightens pattern (`red.400`) only gives white text 3.24:1, failing AA. Used `red.600` instead (darker, breaking the brand-hover directionality convention specifically because `red.400` isn't safe here) ŌĆö 6.12:1. Light-theme hover (`red.700`) mirrors the brand pattern fine at 8.1:1.
- **Real pre-existing bug:** `text.success` on `bg.success-subtle` was **4.08:1, failing AA** ŌĆö shipped in Phase 2 undetected, a symptom of green's steps not being as evenly spaced as red's under the HSL-derived scales (see the OKLCH re-derivation item below). Fixed by moving `text.success` from `green.700` to `green.800` (6.01:1) in both light themes; dark themes were already passing.
- **No neutral/offline status-tone fill existed** (Avatar's offline status dot was reusing `border.strong`, a border-purposed token, as a fill ŌĆö and that reuse also failed WCAG 1.4.11 non-text contrast: 2.74:1 light / 2.36:1 dark against `bg.surface`, needs 3:1). Added `bg.neutral` = `gray.500`, which uniquely passes both light (4.24:1) and dark (3.12:1) against `bg.surface` ŌĆö a single shared value across all 4 themes.

**Post-Phase-3 review pass (auditing for workarounds before Phase 4 turned up two more real gaps):**
- **`text.tertiary` failed AA as text:** it's used both as a non-text icon color (Input's affix icons, 3:1 floor ŌĆö fine at the original `gray.500`) and as literal text color (`Text` component's `color="tertiary"`, `Divider`'s label, `Input`'s placeholder ŌĆö 4.5:1 floor). The original `gray.500` was 4.24:1 light / 3.12:1 dark against `bg.surface`, failing the text bar despite passing the icon bar. Fixed: `gray.600` for light themes (5.60:1), `gray.400` for dark themes (4.82:1) ŌĆö both still correctly more muted than `text.secondary`.
- **Badge's solid variant needed a fuller token set than `text.on-danger` alone:** computed `text.on-info` (white, 6.01:1/4.56:1, passes as one shared value), `text.on-success` and `text.on-warning` (white fails badly on both ŌĆö 2.33ŌĆō3.30:1 ŌĆö used `gray.950` instead, 5.02ŌĆō7.99:1, matching the dark-text precedent from `text.on-brand`'s emerald-dark override), and a dedicated `bg.neutral-strong` (`gray.600`) + `text.on-neutral` (white, 5.60:1) since `bg.neutral` (verified only for the 3:1 non-text case) fell short of 4.5:1 with either white (4.24:1) or dark (3.9:1) text ŌĆö the least accessible possible host for text, sitting near the perceptual midpoint of the gray scale.

**Phase 4 (OKLCH re-derivation, 2026-07-18) ŌĆö full re-verification, since 75 of 77 primitive values changed:**
- Re-ran every pairing above plus a systematic sweep (25 pairings ├Ś 4 themes = 100 checks: all `text.*`/`bg.*` combinations, `bg.neutral`, `border.focus`, `icon.default`, `icon.brand`). 6 categories of regression found and fixed:
  - `text.on-success` / `text.on-warning`: white/gray.950 no longer cleanly passed against the regenerated `green.600`/`amber.600`. Light themes: `bg.success` moved to `green.700` (white text, 6.50:1); `text.on-warning` moved to white (4.78:1 against unchanged `amber.600`). Both tokens are now **theme-specific** (light = white, dark = gray.950) rather than shared.
  - `text.on-danger` / `text.on-info` (dark themes): white against the regenerated `red.500`/`blue.500` dropped to 3.48:1/3.26:1. Fixed by making both **theme-specific** (dark = gray.950: 5.22:1/5.56:1; light keeps white).
  - `bg.danger-hover` (dark themes): now `red.400` (lighter than base, restoring the normal dark-mode "brighten on hover" direction) instead of the Phase 3 `red.600` ŌĆö that darker value was only needed because dark-mode `text.on-danger` used to be white; now that it's gray.950, red.400 works and the more intuitive hover direction is restored.
  - `bg.brand-hover` (purple-dark only): `purple.400` dropped to 3.41:1 with white text. Moved to `purple.600` (darker, 7.37:1) ŌĆö deliberately breaks the "lighten on hover in dark mode" convention here, same class of exception as `bg.danger-hover`'s original Phase 3 fix.
  - `text.link`: purple-dark's `purple.400` (4.13:1) ŌåÆ `purple.300` (5.80:1); emerald-light's `emerald.600` (4.16:1, the brand anchor) ŌåÆ `emerald.700` (6.08:1). The emerald one is a **pre-existing gap unrelated to the regen** ŌĆö same anchor hex before and after Phase 4, it just hadn't been checked as literal text color until this sweep.
  - `border.focus` (emerald-light only): `emerald.500` fell to 2.92:1 against the 3:1 non-text floor. Moved to `emerald.600` (4.16:1).
  - `icon.secondary` (light themes): found failing 3:1 (`gray.400` = 2.32:1) while auditing the remaining icon tokens ŌĆö not a regression (this token isn't referenced by any shipped Phase 3 component yet) but a real gap, fixed pre-emptively. Moved to `gray.500` (3.25:1).
- All fixes verified against the full 100-check sweep plus a Storybook visual pass (both brands ├Ś both modes) before merging. See individual `$description` fields in each `semantic/*.json` file for exact per-token numbers.

**Not yet checked:** the full disabled-state contrast combinations (placeholder text is covered via `text.tertiary`), which typically intentionally sit below AA (expected for disabled states). `icon.disabled` is exempt for the same reason.

## Multi-theme structure going forward

Adding a third brand theme later = one more pair of semantic JSON files (`{brand}-light.json`, `{brand}-dark.json`) referencing a new primitive color scale, following the exact same token names as the existing two. No changes needed to component code, since components should only ever reference semantic tokens, never primitives directly.

## Build pipeline decisions (Phase 2)

Style Dictionary 5's actual behavior diverged from a few of the phase brief's assumptions. Documented here so these don't get "fixed" as bugs in a later session ŌĆö each was audited post-Phase-3 and confirmed to be the correct, standards-conforming response to a real SD5 constraint, not a shortcut.

- **Built-in transforms cover fontFamily/cubicBezier.** SD5 ships `fontFamily/css` and `cubicBezier/css` out of the box ŌĆö no custom transform code needed, contrary to the original assumption. Used directly in `style-dictionary.config.js`'s `CSS_TRANSFORMS`.
- **Primitive files aren't self-namespaced.** `radius.json` and `breakpoint.json` both use bare `sm`/`md`/`lg`/`xl` at the top level, which collide on a naive merge. A custom parser (`dbm/namespace-primitives`) wraps each file's content under the namespace its DTCG aliases already expect (`color.json` ŌåÆ `color`, `spacing.json` ŌåÆ `space`, etc.); `typography.json` and `other.json` need no wrapping since their internal groups are already distinct.
- **The built-in size/rem transform can't handle `clamp()`.** Confirmed empirically ŌĆö it throws `Invalid Number` on fluid typography values. `CSS_TRANSFORMS` deliberately excludes any dimension-unit transform; all dimension `$value`s (spacing, radius, fluid font sizes) are pre-authored as final, correctly-unitted CSS strings (`"0.25rem"`, `"4px"`, `"clamp(1.2rem, 1.186rem + 0.071vw, 1.25rem)"`) at the source, so they pass through the pipeline verbatim instead of being computed by a transform.
- **`shadow.json`'s `_note` key is stripped during parsing.** DTCG parsing expects every leaf to resolve to a `$value`; the human-readable `_note` annotation isn't a token, so the custom parser deletes it, scoped narrowly to that one file/key.
- **Each theme is built as a separate Style Dictionary instance**, not as one config with multiple platforms. The four semantic theme files (`purple-light`, `purple-dark`, `emerald-light`, `emerald-dark`) all use identical top-level token paths (`bg.canvas`, etc.) ŌĆö loading more than one into a single dictionary at once collides. `style-dictionary.config.js` builds 5 instances total: one primitives-only pass plus one per theme, each sourcing the shared primitives plus exactly one semantic file.
